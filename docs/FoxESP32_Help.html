Chapter 1

About Fox ESP32 Firmware

Fox ESP32 Firmware is a single Arduino sketch that runs on a cheap ESP32 dev board and talks to your Flipper Zero over a plain wired UART connection (no BLE pairing, no WiFi bridge between the two devices — just four wires). Your Flipper sends it plain-text commands one line at a time; the ESP32 does the actual WiFi/Bluetooth/internet work and reports back. Four companion Flipper Zero apps give you a menu-driven interface to the firmware so you never have to type raw commands by hand — though you always can, via the Terminal.

For authorized security testing and research only. WiFi/BLE reconnaissance and attack commands are gated behind a single Attacks toggle (on by default, matching how most similar tools ship) — see Safety, Legal & the Attacks Toggle. Know your local regulations before transmitting anything.

The five pieces of the project:

Fox ESP32 Firmware
Runs on the ESP32
The actual WiFi/BLE/HTTP/Discord/scripting engine. One firmware image, flashed once.
Fox ESP32 Detector
Flipper app
Diagnostic tool — tells you which GPIO pins and baud rate your ESP32 is actually wired/talking on.
Fox ESP32 Commander
Flipper app
The main control app — WiFi, Bluetooth, Tag Detection, Scripts, Settings, Terminal.
Fox Chat
Flipper app
Post to and read from a Discord channel through the ESP32's internet connection.
Fox Portal
Flipper app
A configurable, consent-based lead-capture WiFi captive portal with QR join and CSV logging.

What you can do with it:

  • WiFi reconnaissance — scan access points and stations, monitor signal strength, sniff beacons/deauths/probes/raw frames, flag likely PMKID/SAE handshake activity, export a real .pcap file straight to Wireshark, or log a GPS-tagged wardrive to CSV.
  • WiFi attacks (gated) — broadcast or targeted deauth, custom or random beacon spam, probe flood, a joke SSID list, malformed-frame bursts, spoofed channel-switch/power-save frames, and a plain TCP port scanner.
  • MAC address tools — randomize, clone a specific address, clone the selected AP's address, or reset to the factory MAC. Software-only, no extra hardware.
  • Bluetooth LE tools — a general BLE bridge (built for a Chameleon Ultra but usable for general scan/connect/notify work), a gated multi-vendor pairing-popup spam tool, and non-gated item-tracker detection (Apple Find My, Samsung SmartTag, Tile).
  • FoxChat — post messages to a Discord channel and read the most recent ones back, straight from your Flipper's screen, with basic outgoing content filtering.
  • FoxPortal — stand up an open WiFi network with a custom enquiry-form landing page, log submissions to a dated CSV, and hand out a join QR code — fully reconfigurable fields/copy/HTML without a reflash.
  • FoxScript — a small embedded scripting language (14 native modules covering WiFi, HTTP, BLE, GPIO, storage, and more) for automating multi-step actions without a computer involved.
  • A FlipperHTTP-compatible internet bridge — bracket-command GET/POST/PUT/PATCH/DELETE (including binary-safe base64 variants), a WebSocket client, and simple JSON field extraction — usable directly from Terminal or by other Flipper apps that already speak FlipperHTTP.
  • Optional hardware add-ons — RFID (PN532), Sub-GHz (CC1101), IR, and GPS support, each gated behind a build-time flag for boards that have the extra breakout wired up.

Chapter 2

Web Installer

The ESP32 side of this project is flashed from your browser — no Arduino IDE, no esptool command line, nothing to install. You only need a USB cable and a Chrome- or Edge-based desktop browser (Web Serial support is required; Safari and Firefox cannot do this).

⚡ Open ESP32 Web Flasher →

Hosted alongside the Flipper firmware flasher on the FoxFW GitHub Pages site.

Keep the browser tab in focus while flashing. The Web Serial connection and the flash transfer both run in that tab — switching away, closing the laptop lid, or letting the system sleep mid-flash can interrupt the transfer.

Choosing your board

The flasher's Model dropdown currently lists two board targets. Pick the one that matches your actual ESP32 hardware — flashing the wrong image for your chip family will not work (see the chip-mismatch protection below, which exists specifically to catch this before it happens).

Board entryChip familyBootloader address
Fox ESP32 (Classic)ESP320x1000
Fox ESP32-S3ESP32-S30x0

Both boards flash the same four files — Bootloader, Partitions, Boot App0, and Firmware — at the same addresses except for the bootloader itself, whose address differs by chip family (classic ESP32 boots its bootloader from a different flash offset than S-series/C-series chips do). If the board's firmware set isn't hosted online yet, the flasher tells you so in the console and lets you pick each of the four .bin files manually from your computer instead.

ESP32-C3/C6/C2/C5/H2 and other variants aren't listed as ready-made board entries yet, but the underlying firmware is expected to work on them (same Arduino APIs via arduino-esp32's RISC-V toolchain support) — see Hardware Setup for what changes per chip. If you compile for one of these yourself, you can still flash it with this tool using the manual file picker; just pick the correct bootloader address for your chip.

Step by step

  1. Open the Web Flasher link above in Chrome or Edge on desktop.
  2. Pick your board in the Model dropdown.
  3. If prompted, choose each of the four firmware files manually (Bootloader / Partitions / Boot App0 / Firmware) — these are the files you built with Arduino IDE's Sketch → Export Compiled Binary, see Hardware Setup.
  4. Plug your ESP32 into your computer via USB and set the Baud rate dropdown (defaults to 115200 — leave this as-is unless you know you need otherwise).
  5. Click Connect and pick your device from the browser's own serial port picker.
  6. Once connected, confirm the detected chip matches your selected board (the flasher checks this automatically).
  7. Click Program. A progress bar tracks each of the four files as they're written.
  8. When it finishes, the ESP32 resets automatically. The flasher then locks Program/Erase again until you disconnect and reconnect — this is expected, not an error.

Chip-mismatch protection

When you click Connect, the flasher reads back the ESP32's own self-reported chip identity and compares it against the board you selected. If they don't match, the status pill turns red, the console explains exactly what was selected versus what was detected, and the Program button is disabled so you can't accidentally write a classic-ESP32 image to an S3 board (or vice versa) — mismatched firmware for the wrong chip family simply won't boot. Erase stays available even on a mismatch, since erasing doesn't write anything board-specific and can help you start clean if you picked wrong.

Erasing the flash

The Erase button wipes the entire flash chip. You'll get a confirmation prompt first. This is useful if a previous flash left the board in a bad state, or before switching a board between firmware projects.

Flashing via Flipper Zero (no PC required)

If you already have FoxFW installed on your Flipper, you can flash the ESP32 directly from the device — no browser, no USB cable to a computer, no files to download. The firmware for every supported board is bundled into FoxFW and deployed to your SD card automatically during the standard qFlipper install.

  1. Connect your ESP32 to the Flipper Zero GPIO header: TX (GPIO1) → Pin 14, RX (GPIO3) → Pin 13, GND → GND, 3.3V → 3.3V.
  2. On your Flipper go to Apps → Fox → Fox ESP32 Flasher.
  3. The app detects your connected ESP32 automatically. If nothing is found, check your wiring and try again.
  4. Select Flash Firmware, then choose your board type from the list.
  5. Confirm the firmware files shown and press Flash. The app writes the bootloader, partition table, and firmware in a single pass and shows a progress bar.
  6. Press Done when complete — the ESP32 resets automatically.
Recommended for re-flashing in the field. Once FoxFW is on your Flipper, the Fox ESP32 Flasher is the fastest way to update or recover an ESP32 without needing a computer nearby.

Chapter 3

Hardware Setup

Wiring the UART link

The Flipper and the ESP32 talk over a plain 4-wire UART connection — no special adapter needed, just jumper wires from the Flipper's GPIO header to the ESP32's RX/TX/GND/5V (or 3V3) pins. Every Flipper-side app in this project (Detector, Commander, Fox Chat, Fox Portal) probes the same two standard pin pairs automatically at boot, so as long as you wire into one of these two pairs, every app will find your ESP32 without any manual configuration:

PairFlipper GPIO pinsInternal name
Pair 113 / 14USART
Pair 215 / 16LPUART

All communication runs at a fixed 115200 baud.

Don't know which pins you used, or whether your wiring even works? That's exactly what Fox ESP32 Detector is for — it tests both pairs automatically and tells you which one is live.

Board differences

Fox ESP32 Firmware supports the classic ESP32 and its variants, with real differences in what each one can do:

ESP32 (Classic) / ESP32-S3
Full support — WiFi, raw 802.11 frame injection (needed for deauth/beacon-spam/etc.), and Bluetooth LE all work. These are the two boards with ready-made Web Installer entries today.
ESP32-C3 / C6
Expected to work unmodified — same Arduino APIs, just compiled for the RISC-V toolchain arduino-esp32 provides for these chips. No ready-made Web Installer entry yet; compile and flash manually.
ESP32-C5
Needs a recent-enough arduino-esp32 core. Raw-injection support for attack commands hasn't been independently confirmed on this chip.
ESP32-S2 No BLE
This chip has no Bluetooth radio at all — a hardware limitation, not a firmware choice. WiFi, HTTP, and FoxScript all work normally. Every Bluetooth-related command (the BLE bridge, BLE spam, Tag Detection) replies with a plain error instead of failing silently, so it's always obvious when you're on an S2 board.
Applies to: the Bluetooth menu and Tag Detection menu in Fox ESP32 Commander

Optional add-on hardware

Four extra hardware peripherals can be wired to the ESP32 for expanded functionality. Each one is a separate breakout board you add yourself — none of them ship as part of the base kit — and each is controlled by its own build-time flag in the firmware, off by default. If you want one of these, you (or whoever builds your firmware image) needs to turn the flag on and recompile; it can't be toggled at runtime like the Attacks setting can.

PeripheralChip / interfaceWiringWhat it adds
RFID PN532 breakout, I2C IRQ → GPIO 25, Reset → GPIO 26 (SDA/SCL use the ESP32's default I2C pins) Mifare Classic scan / read / write
Sub-GHz CC1101 breakout, SPI SCK → 18, MISO → 19, MOSI → 23, CS → 27, GDO0 → 32 Raw sub-GHz RX/TX (300–928 MHz) — no protocol decoding, raw capture/replay only
IR IR LED + demodulating receiver Send → GPIO 4, Receive → GPIO 5 Send/receive/decode across dozens of IR protocols, plus a small TV-power-toggle sweep
GPS Any NMEA 0183 GPS breakout (e.g. NEO-6M/8M) RX → GPIO 16, TX → GPIO 17, 9600 baud (UART2) Live fix/position/track data, POI marking, and GPS-tagged WiFi wardriving

With a peripheral's flag left off, every command for that hardware replies with a specific error (ERROR:NORFID, ERROR:NOSUBGHZ, ERROR:NOIR, ERROR:NOGPS) rather than doing nothing silently, so it's always clear whether a missing feature is "not wired up" versus "not working."

Chapter 4

Fox ESP32 Detector

A single-purpose diagnostic app: it tells you whether anything is actually talking on your Flipper's GPIO header, and if so, which pins, which baud rate, and — where it can tell — which firmware. There's no menu, no configuration, and nothing is saved. Point it at a freshly wired board when you're not sure the wiring is right, or when Fox ESP32 Commander reports "not detected" and you want to know why.

How it works

Press Scan and watch a live progress bar work through every combination of the two Flipper pin pairs and nine common baud rates — 18 combinations total. For each one, it tries up to four detection tiers in order, stopping at the first one that answers:

Tier 1Listen for GPS
(NMEA sentences, no command sent)
→
Tier 2Send AT
ESP-AT firmware check
→
Tier 3Send info
Known non-AT firmware (Bruce, Fox ESP32 Firmware)
→
Tier 4Send ?
Generic fallback, requires a repeatable response

Baud rates tested: 9600, 19200, 38400, 57600, 74880, 115200, 230400, 460800, 921600. A full scan of a board with nothing connected takes roughly 30 seconds, since every combination has to time out before moving on.

What it can identify today: Fox ESP32 Firmware itself (matches this project's own self-identification string), and Bruce firmware (a different, unrelated ESP32 project) as a bonus. Marauder doesn't have a safe self-identify command, so a Marauder-flashed board will show up as "Unidentified" rather than being named.
Scope note — what this tool does not detect: a raw, unflashed ESP32 chip's bootloader (that needs two additional wires and a different protocol entirely), I2C/SPI sensor breakouts, or anything that doesn't speak line-based serial commands. This is purely a "does something on this GPIO pair understand my Flipper's serial protocol" check.

Chapter 5

Fox ESP32 Commander

The main control app for the firmware — WiFi, Bluetooth, item-tracker detection, automation scripts, the safety toggle, and a raw Terminal, all in one menu tree. This is the app you'll spend the most time in.

First run & detection

Opening the app always follows the same sequence:

1. SplashFox logo, ~2s
→
2. DetectProbes both pin pairs for the firmware's self-ID
→
3a. FoundStraight to the main menu
→
3b. Not foundRetry screen

If nothing answers on either pin pair, you'll see a short message — "Fox ESP32 Firmware required on ESP32 connected via GPIO." — with two options: Retry (probes both pin pairs again, useful if you just plugged something in) and Settings (opens Connect Settings, below).

Connect Settings
Reached only from the Retry screen's Settings button. An arrow-adjust screen with a Pins row (13/14 or 15/16) and a Baud row (115200 only today — the row is built to support more rates later). A Retry row re-probes using exactly the pair you've selected, rather than trying both. Use this if you know exactly which pins you wired and want to skip the automatic dual-pair probe.
Not detecting at all? Run Fox ESP32 Detector first to confirm your wiring and baud rate independently of this app.

Main menu:

Fox ESP32 Commander ├── WiFi ├── Bluetooth ├── Tag Detection ├── Scripts ├── Settings └── Terminal

WiFi menu

WiFi ├── Connect / Connection (label changes based on live connection state) │ not connected → "Connect" → network list → password entry │ connected → "Connection" submenu: │ ├── Disconnect │ ├── Status (SSID + IP address) │ ├── Connect to Network (reopens the list to switch networks) │ └── Forget Network ├── My IP ├── HTTP Request │ ├── GET (one prompt: URL) │ └── POST (two prompts: URL, then body) ├── Recon │ ├── Scan APs │ ├── Scan Stations │ ├── Select AP (target for signal monitor / attacks) │ ├── Signal Monitor (10s live RSSI on the selected AP) │ ├── Packet Count (5s promiscuous frame count) │ ├── Wardrive (GPS-tagged AP scan → dated CSV) │ └── Packet Capture (15s raw capture → .pcap file) └── Attacks Gated ├── Select AP ├── Select Station ├── Deauth (Broadcast) ├── Deauth (Targeted) ├── Beacon Spam (Random) ├── Beacon Spam (Custom) └── Probe Flood

Connect / Connection

The network list merges a live scan with your saved networks (saved ones shown first), as rounded Fox-styled rows showing SSID, signal strength, security type, and whether it's saved. Selecting a saved network reconnects immediately. Selecting a new one prompts for a password (leave blank for open networks) and saves it for next time. Once connected, the same menu item relabels itself Connection and opens a submenu instead: Disconnect, Status (current SSID + IP), Connect to Network (switch to a different saved/new network), and Forget Network (removes the saved password and disconnects).

This is the menu that drives your Flipper's own status bar WiFi icon. Every Connect/Disconnect/Forget/Status action here updates a small status file on your SD card that the Flipper's desktop polls — see Status Bar WiFi Icon. Fox Chat and Fox Portal don't touch this file — they just use whatever WiFi connection is already active.

My IP & HTTP Request

My IP fetches your ESP32's public internet-facing IP address and prints it to Terminal. HTTP Request gives you a quick GET (just a URL prompt) or POST (URL, then a body prompt) without needing to type bracket-command syntax yourself — the raw response is shown in Terminal either way. This is deliberately just GET/POST with no custom headers or other verbs; Terminal is there for anything more advanced (see the full HTTP command reference).

Recon

Scan APs and Scan Stations populate the network/station lists used elsewhere in this menu. Signal Monitor and Packet Count are quick, bounded checks (10 and 5 seconds respectively) that stream straight to Terminal. Wardrive and Packet Capture are the two data-logging tools:

Wardrive
Scans nearby access points and tags each one with your current GPS fix (requires the GPS peripheral, see Hardware Setup). Appends to a dated CSV on your SD card, one file per day.
File: /ext/apps_data/fox_esp32/wardrive/wardrive_<YYYY-MM-DD>.csv
Columns: bssid, ssid, latitude, longitude, rssi, channel, encryption_type, timestamp
Packet Capture
Captures raw 802.11 frames for 15 seconds, hopping channels 1–11 every 400ms, and saves a real, standard .pcap file — opens directly in Wireshark, no conversion needed.
File: /ext/apps_data/fox_esp32/captures/capture_<timestamp>.pcap

Attacks Gated

Every item in this submenu requires Attacks to be turned on in Settings — with it off, every action here replies with a plain "disabled" message instead of doing anything. All attack primitives run for a fixed 5-second burst and then stop on their own; no separate stop button exists because none is needed.

  • Deauth (Broadcast) — deauthenticates every client of the currently selected AP.
  • Deauth (Targeted) — deauthenticates one specific station (pick it via Select Station first).
  • Beacon Spam (Random) — floods the air with 8 randomly-named fake networks.
  • Beacon Spam (Custom) — same idea, but you type a comma-separated list of SSIDs to cycle through.
  • Probe Flood — sends a burst of probe requests.
Select AP / Select Station first. Deauth and several other attacks act on whichever AP/station you last selected via Recon's scan screens — if you haven't picked one, targeted actions will tell you so rather than doing nothing silently.

Bluetooth menu

This menu is BLE spam only — five gated items, each cycling a proximity-pairing-shaped advertisement for 8 seconds to trigger the matching OS's pairing popup nearby.

Bluetooth ├── iOS ├── Windows ├── Samsung ├── Android └── All (rotates through all four)
Not available on ESP32-S2 boards (no BLE radio). This menu is unrelated to the general-purpose BLE bridge command set (BLEINIT, BLESCAN, BLECONN, etc.) built into the firmware for talking to other BLE devices — those commands are reachable from Terminal but don't have a dedicated menu screen in this app.

Tag Detection menu

Tag Detection ├── AirTag / Find My ├── Samsung SmartTag └── Tile

Each item runs a bounded 8-second BLE scan for that tracker family's broadcast signature and prints any matches straight to Terminal — mac address, signal strength, and (for Find My only) a decoded battery hint. This is not gated by the Attacks toggle — detecting nearby trackers is passive listening, not an attack.

This is a snapshot, not a stalking detector. A match means "a tag matching this family's broadcast format is nearby right now" — it does not mean the tag has been following you, and repeated scans over time aren't tracked or correlated. AirTag/Find My detection matches Apple's shared "offline finding" broadcast format, which every Find My accessory uses (not just AirTags specifically) — the protocol itself can't distinguish which exact product sent it.

Scripts menu

Lists every script currently saved on the ESP32, plus a trailing New Script item. Selecting an existing script opens Run / Show / Delete (deliberately not run-on-select, so Delete is never one accidental press away). New Script prompts for a name, then the script's source code, and saves it.

Typing long scripts on-device is slow. For anything more than a few lines, write your script as a single line and send it via Terminal using the raw SCRIPTSAVE: command instead — see FoxScript for the full language reference.

Settings & Terminal

Settings — Attacks: ON/OFF
A single arrow-adjustable row. Changes apply immediately over serial — there's no separate save step. This screen always queries the firmware's real, persisted value when you open it, so it never drifts out of sync with what the ESP32 actually has stored. This is different from Connect Settings (pin/baud override), which only appears from the not-detected Retry screen.
Terminal
Raw command access to the firmware, and also where most menu actions log their results — a single running transcript of everything that's happened. Type any command from the full command reference directly here.

Chapter 6

Fox Chat

Post to and read from a Discord text channel, straight from your Flipper's screen, using your ESP32's internet connection. You'll need an active WiFi connection (set up via Fox ESP32 Commander's WiFi menu) and a Discord bot pointed at a channel.

Fox Chat ├── Post Message ├── Read Messages └── Bot Settings
Post Message
Prompts for your message text, then sends it prefixed with your Flipper's own device name — e.g. MyFlipper: hello — so anyone reading the channel can tell which device posted it (Discord bot accounts don't have a per-message display-name feature outside of webhooks, so this is the practical equivalent). If the message is blocked by the content filter you'll see "Blocked - message flagged by content filter"; if you're posting too quickly you'll see "Too soon - wait a few seconds and try again."
Read Messages
Fetches the most recent messages and shows them as a scrollable list of two-line buttons — select one to see the full text and the exact time it was sent. Wherever a message has that DeviceName: prefix (i.e. anything posted through Fox Chat, from any Flipper), the device name renders in bold with the rest of the message following normally.
Bot Settings
Prompts for your bot token, then your channel ID, and saves both to the ESP32 (persists across reboots — you only need to do this once per bot/channel).
Works out of the box without setup. If you post or read before ever visiting Bot Settings, Fox Chat automatically falls back to a pre-configured demo bot/channel baked into the app, so you can try the feature immediately. Run Bot Settings whenever you want to point it at your own bot and channel instead — once you do, your own credentials take over.

Setting up your own Discord bot

  1. Go to the Discord Developer Portal, log in, click New Application, and give it a name.
  2. Open the Bot tab and copy the bot token (treat this like a password — anyone with it can post as your bot).
  3. On the same tab, turn on Message Content Intent under Privileged Gateway Intents — required for reading message text.
  4. Go to OAuth2 → URL Generator, check the bot scope, then check only Send Messages and Read Message History under bot permissions. Copy the generated URL.
  5. Paste that URL into your browser, pick your server, and click Authorize.
  6. In Discord, enable Developer Mode (User Settings → Advanced), then right-click your target channel and choose Copy Channel ID.
  7. Enter the bot token and channel ID into Fox Chat's Bot Settings screen.

Content moderation — what's protected and what isn't

  • Every outgoing message is checked against a built-in word/phrase filter before sending (toggle in Fox ESP32 Commander's Settings — on by default). It's plain word-matching, not spacing- or leetspeak-aware, so it can occasionally false-positive on an innocent word that contains a blocked substring.
  • Mentions are always sent inert — @everyone, @here, and role pings render as plain text and can't actually ping anyone, regardless of what you type.
  • A short minimum gap is enforced between posts to prevent accidental flooding.
  • Discord's own AutoMod does not see bot-posted messages — AutoMod only scans human-account messages, so whatever protections your server has configured there won't apply here. The built-in filter above is the only thing standing between what you type and the channel.
  • Anything genuinely abusive or illegal posted through the bot is still a Discord Terms of Service violation on the bot account (and, by extension, whoever's Discord developer account created it).

Chapter 7

Fox Portal

Stands up an open WiFi network with a small enquiry-form landing page — anyone who joins sees your page, fills it in, and gets a confirmation screen, with every submission logged to a dated CSV file. Unlike a typical "evil portal," this one is built for consent-based lead capture: the visitor is knowingly opting in and the confirmation page shows them exactly what was recorded, not a credential phishing trap.

Fox Portal ├── Start ├── Stop ├── Edit Input Names └── Show QR
Start
Prompts for an SSID (leave blank to use the default). Before starting, it re-reads your input names and page files fresh from the SD card every single time, so any edits you've made — from the app itself or by hand on a computer — are always picked up. The portal then goes live as an open WiFi network with a captive-portal redirect.
Stop
Shuts down the WiFi network and web server.
Edit Input Names
Add or remove the field names your form collects, right from the Flipper's screen — no computer needed. At least one field is always required; the list can never be emptied entirely.
Show QR
Displays a scannable WiFi-join QR code for the currently active portal SSID, so visitors can join with a camera scan instead of typing a network name. The portal network is always open (no password).

Customizing the pages — start.html, finish.html, inputnames.txt

These three plain files live on your Flipper's own SD card and are the real source of truth for what visitors see — nothing here is compiled into the ESP32 firmware, so editing them never requires a reflash.

FilePathWhat it controls
start.html/ext/apps_data/fox_portal/The enquiry-form page visitors see first
finish.htmlThe confirmation page after submitting
inputnames.txtOne field name per line — doubles as the CSV column headers

Open and edit these directly — via qFlipper's file browser, the Flipper Mobile App, or by pulling the SD card into a computer. In start.html, each form field needs <input name="<fieldname>"> matching a line in inputnames.txt. In finish.html, use {{FIELD:<fieldname>}} anywhere you want to echo back what the visitor submitted for that field.

Starter templates are included automatically. A ready-made example set — a plain "Get In Touch" page with name and phone fields, styled with a dark background and orange accent — is bundled into the firmware and deployed to /ext/apps_data/fox_portal/ on your SD card during the standard qFlipper install. No manual copying required. Edit the files from there to customise your portal.
Field names must match. If you rename a field in inputnames.txt without updating start.html's matching name="..." attribute, nothing crashes — that submission's cell for the mismatched field just logs empty, and everything else you did capture is still saved.

Each HTML file has a 1024-byte size limit once flattened to a single line (a transfer-safety limit, not an HTML restriction) — the included starter templates use under 1000 bytes each, leaving room to adjust copy or light styling.

Submission logging

Every submission is appended to a CSV file on the ESP32, named after the day it was captured (using your Flipper's own clock, since the ESP32 has no clock of its own). Every configured field gets a column on every row, even if a particular visitor's page didn't collect it that time — a missing value just logs as an empty cell rather than dropping the row.

Chapter 8

Status Bar WiFi Icon

Your Flipper's own desktop status bar shows a small WiFi icon, immediately to the left of the battery indicator, reflecting your ESP32's connection state as last reported by Fox ESP32 Commander's WiFi menu.

StateAppearance
ConnectedThree bold, solid ascending bars
Disconnected / idleThe same three bars, drawn as thin lines instead of solid — still visible, not an error indicator
How this actually works. The Flipper's desktop can't talk to the ESP32 directly while a Fox app has the serial connection open, so instead it polls a small status file on your SD card every two seconds. Only Fox ESP32 Commander's WiFi menu writes that file — Fox Chat and Fox Portal just use whatever WiFi connection already exists and never change it, so the icon only ever reflects actions taken through Commander's WiFi menu, updated within a couple of seconds of you taking them.

The icon stays visible even when disconnected on purpose — if it vanished entirely, someone who hasn't discovered the Fox ESP32 apps yet would have no hint that this feature exists at all.

Full Command Reference

Every command the firmware understands, sent over serial at 115200 baud. You'll rarely need to type these by hand — Fox ESP32 Commander, Fox Chat, and Fox Portal send them for you — but they're all reachable from Commander's Terminal screen if you want raw access, and knowing them helps make sense of what each menu action is actually doing.

Liveness check. AT replies OK. info replies Fox ESP32 Firmware — this exact string is what Fox ESP32 Detector and every app's detection screen look for to confirm they're talking to this firmware and not something else.

WiFi reconnaissance

FoxScript

SCRIPTLISTLists saved scripts

FoxScript is a small purpose-built scripting language (not JavaScript, though the syntax looks familiar) — variables, if/else, repeat(n) loops, numbers, strings, arithmetic/comparison/logical operators, and function calls including dotted native calls. No user-defined functions, arrays, objects, or while/for. Scripts run straight off the ESP32's own storage — no SD card, no reflash. 

let name = "world";
print("hello " + name);

if (wifi.connect("MySSID", "MyPassword")) {
  let body = http.get("http://api.ipify.org");
  print("my IP is " + body);
} else {
  print("wifi connect failed");
}

repeat (3) {
  print("free heap: " + device.freeHeap());
  delay(1000);
}

repeat is capped at 1000 iterations, and a whole script run is capped at 200,000 evaluation steps as a safety net against a mistyped infinite loop.

Chapter 10

Safety, Legal & the Attacks Toggle

For authorized security testing and research only. WiFi deauthentication, beacon spam, probe flooding, BLE proximity-pairing spam, and similar tools in this firmware can disrupt networks and devices you don't own. Many jurisdictions treat this as illegal regardless of intent, even against your own equipment if it also affects a neighbor's network. Only point these at networks and devices you own or have explicit, documented permission to test.

The Attacks toggle

All WiFi attacks, BLE spam, the TCP port scanner, and Fox Portal are gated behind one setting, checked with SETTINGS and changed with SETTINGS:ATTACKS:ON / SETTINGS:ATTACKS:OFF — reachable from Fox ESP32 Commander's Settings screen as a single "Attacks: ON/OFF" row that applies immediately.

On by default. A fresh device ships with attacks enabled — matching how comparable tools like ESP32Marauder behave out of the box. If you'd rather these stay off until you deliberately turn them on, visit Settings right after your first flash and switch it off. The setting is saved to the ESP32's flash and survives reboots, so you only need to set it once.

While attacks are off, every gated command replies plainly with ERROR:DISABLED — nothing is hidden or silently ignored.

Built-in self-limits

Every attack primitive (deauth, beacon spam, probe flood, BLE spam, the novelty attacks, the port scanner) runs for a fixed, short burst — 5 seconds for WiFi/port-scan attacks, 8 seconds for BLE spam — and then stops on its own, replying ATTACKDONE. There's no "run forever" mode and no separate stop command to remember; re-run the command if you want another burst.

Content moderation in Fox Chat

Outgoing Discord messages are checked against a built-in word filter before sending (toggle: SETTINGS:PROFANITY:ON/OFF, on by default), and every post is sent with mentions disabled so @everyone/@here/role pings can never actually ping anyone. See Chapter 6 for the full detail, including what this protection does not cover (Discord's own AutoMod doesn't scan bot-posted messages).

Fox Portal is consent-based

Fox Portal is a lead-capture tool, not a credential-phishing "evil portal" — visitors knowingly join an open network and see exactly what they're submitting, with a confirmation page that echoes it back. Don't use it to impersonate another organization's network or to collect information from people who haven't knowingly agreed to give it.

Chapter 11

Known Limitations

An honest list of rough edges, so nothing here catches you by surprise. None of these are dealbreakers for normal use — most are "this hasn't been battle-tested against every possible target/accessory yet," not "this is broken."

Firmware

  • HTTPS requests don't validate the server's certificate — fine for casual API calls, not for anything sensitive.
  • The JSON field extractor only understands flat objects plus one level of nested headers — not deeply nested or general array-valued JSON (beyond the dedicated [PARSE/ARRAY] command).
  • FoxScript is a small custom language, not real JavaScript — no user-defined functions, arrays, objects, or while/for loops.
  • WIFISNIFF:PMKID/SAE flag frame shapes (an EAPOL frame was seen; a SAE Commit frame was seen) rather than performing full protocol extraction.
  • RFIDEMULATE is not implemented — everything else in the RFID/Sub-GHz/IR/GPS command families is real, working driver code.
  • The BLE spam payloads match each vendor's proximity-pairing packet framing (AD structure, company ID, UUID) correctly, but the exact inner payload bytes beyond that framing are a best-effort approximation.

Fox ESP32 Commander

  • The AP/station scan-result list re-scans every time you open it — there's no caching between visits.
  • SSIDs, passwords, URLs, and script text sent to the firmware aren't JSON-escaped — a literal quote or backslash in one of these can break the request.
  • Custom-SSID beacon spam and targeted deauth only check that the field isn't empty, not that it's well-formed.
  • The Connect Settings baud-rate row only offers 115200 today (built to support more rates later).

Fox ESP32 Detector

  • Scans the two standard pin pairs (13/14, 15/16) across 9 baud rates each — a full scan with nothing connected takes about 30 seconds.
  • Only recognizes a small, growing list of non-AT firmware signatures by name (currently this firmware and Bruce) — anything else that responds is reported generically as "Unidentified" rather than misidentified.
  • Doesn't detect raw unflashed ESP32 chips, I2C/SPI sensor modules, or Flipper's own built-in IR — by design, out of scope for this tool.

Tag Detection

  • A snapshot tool, not a stalking/tracking detector — a positive result means a tag is broadcasting nearby right now, not that anything has been following you over time.
  • Find My detection catches any Find My-network accessory (AirTag, Chipolo, Pebblebee, etc.), not AirTags specifically — the broadcast protocol itself doesn't distinguish between them.

General

  • ESP32-S2 boards have no Bluetooth radio at all — Bluetooth menu, Tag Detection, and every BLE command are unavailable on that chip.
  • The status bar WiFi icon reflects only the last action taken through Fox ESP32 Commander's WiFi menu — it does not live-poll the ESP32 and doesn't reflect WiFi state from Fox Chat or Fox Portal, since those apps never change the connection themselves.

Support & Troubleshooting

Nothing responds / apps say "not detected"

  1. Run Fox ESP32 Detector first — it's built exactly for this. It'll tell you which pin pair and baud rate (if any) something is answering on.
  2. Double-check wiring against Chapter 3 — pins 13/14 (USART) or 15/16 (LPUART), plus a shared ground.
  3. Confirm the ESP32 is actually powered — a USB cable that only carries data and not power, or a loose connection, is a common culprit.
  4. Make sure the firmware was actually flashed successfully — re-run the Web Installer's Erase, then Program again.
  5. If one app detects it but another doesn't, try Connect Settings (reachable from that app's Retry screen) to manually pick the pin pair Detector identified.

qFlipper "Unknown Error" during firmware install

If qFlipper stops at around 23% with "Verifying existing files…" and then shows Unknown Error / "Operation timeout (generic)" on Storage Md5Sum resources.ths, the most common cause is a stale Update folder left on the SD card from a previous install attempt. The checksum qFlipper computed no longer matches what is on the card.

Fix: delete the /Update folder from the root of your SD card (accessible via qFlipper's file browser or the Flipper Mobile App), then try installing again. If you cannot easily locate the folder, formatting the SD card achieves the same result — your FoxFW apps and settings will be re-deployed on the next successful install.
